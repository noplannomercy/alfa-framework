# ALFA (AI/LLM Framework for Adoption) v1

> 기업이 LLM을 조직에 도입할 때 거치는 전체 프로세스를 체계화한 프레임워크.
> ALFA → IRM → 구현/운영. ALFA는 "도입", IRM은 "설계", 이후가 "실행"이다.
> 최종 갱신: 2026-03 v1

---

## 왜 ALFA인가

LLM 도입 방법론이 아직 없다. 클라우드 도입(AWS CAF), AI 위험관리(NIST AI RMF), AI 관리체계(ISO 42001)는 있지만, **"LLM을 조직에 도입하는 A to Z"를 단계별로 안내하는 실행 프레임워크**는 존재하지 않는다.

그 결과:
- 기업의 42%가 AI 이니셔티브를 폐기 (S&P Global 2025, 전년 17%에서 급증)
- CEO의 56%가 AI에서 "아무것도 못 얻었다" (PwC 2026)
- 파일럿의 54%만 프로덕션 전환에 성공 (Deloitte 2026)
- GenAI 구현의 대다수가 초기 기대치에 미달 (Deloitte 2026 — 기대치 정의에 따라 편차 있음)

ALFA는 이 실패를 방지하기 위해 설계되었다.

---

## 설계 근거

ALFA는 단독 설계가 아니다. 다음 표준/연구를 참조하여 구성했다.

| 참조 | ALFA 반영 지점 | 핵심 차용 |
|------|-------------|---------|
| **NIST AI RMF 1.0** | Phase 2 GOVERN, Phase 6 OPERATE | Govern→Map→Measure→Manage 사이클. 거버넌스 우선 원칙 |
| **ISO/IEC 42001:2023** | Phase 2 GOVERN | PDCA 기반 AI 관리체계. 인증 가능 프레임워크 |
| **AWS CAF-AI** | Phase 1 ASSESS, 전체 구조 | Envision→Align→Launch→Scale 4단계. 6개 관점 |
| **Gartner AI TRiSM** | Phase 2 GOVERN, Phase 6 OPERATE | AI 거버넌스→런타임 검사→정보 거버넌스→인프라 4계층 |
| **BCG 10-20-70 Rule** | People 분산 원칙 | 알고리즘 10%, 데이터/기술 20%, 사람/프로세스/문화 70% |
| **금감원 AI 가이드라인** | Phase 2 GOVERN (금융 특화) | 7대 원칙: 거버넌스, 합법성, 보조수단성, 신뢰성, 금융안정성, 신의성실, 보안성 |
| **인공지능기본법** (2026.01 시행) | Phase 2 GOVERN | 5대 의무: 투명성, 안전성, 고영향AI, 영향평가, 국내대리인 |
| **MAS AI Risk Management** | Phase 2 GOVERN (금융 특화) | 범위/감독→위험관리→라이프사이클→인에이블러 |
| **EU AI Act** | Phase 2 GOVERN | 리스크 등급별 규제. 고위험 시스템(신용평가, 사기탐지) 특별 의무 |
| **McKinsey State of AI 2025** | Phase 0 WHY | 가치 창출은 코어 비즈니스 기능 집중 시 발생 |
| **Prosci ADKAR** | People 분산 원칙 | Awareness→Desire는 변화 이전에 구축 |
| **Google Cloud AI Readiness** | Phase 1 ASSESS, People 분산 | Strategy/Data/People/Technology를 기술 배포 이전에 평가 |

## ALFA 원칙

**1. 검증 전 과투자 금지**
API로 시작하고, 데이터로 판단하고, 필요 시 승격한다. 파일럿도 안 해본 단계에서 GPU를 사지 않는다.

```
API only → API + 필요시 자체 호스팅 → 최적화
(검증)     (확산)                    (운영)
```

**2. 각 Phase에 Owner를 지정한다**
소유자 없는 Phase는 실행되지 않는다. Phase별 RACI를 정의한다.

**3. People은 Phase가 아니라 모든 Phase에 내장된다**
BCG 연구에 따르면 AI 투자의 70%가 사람/프로세스/문화에 가야 한다. 교육과 변화관리를 별도 Phase로 분리하면 "나중에 하자"가 되고, 결국 실패한다. 모든 Phase에 People 활동을 내장한다.

## 전체 구조

```
ALFA Framework v1
├── Phase 0: WHY — 목적 정의
├── Phase 1: ASSESS — 조직 준비도 평가
├── Phase 2: GOVERN — 거버넌스 & 정책
├── Phase 3: PLATFORM — 인프라 & 기술 기반
├── Phase 4: PILOT — 파일럿 실행 (★ IRM 등장)
├── Phase 5: SCALE — 프로덕션 전환 & 확산
└── Phase 6: OPERATE — 운영 & 지속 개선
    └── 순환: 새 유즈케이스 → Phase 4, 규제 변경 → Phase 2
```

## Phase 0: WHY — 목적 정의

**목적:** LLM 도입의 비즈니스 근거를 확립한다. 기술이 아닌 문제에서 시작한다.

> "챗봇 만들자"는 목적이 아니다. "고객 상담 응대 시간을 50% 줄이자"가 목적이다.
> — McKinsey: AI 가치 창출은 코어 비즈니스 기능 집중 시 발생

| # | 항목 | 산출물 |
|---|------|-------|
| 1 | 비즈니스 문제 정의 | 문제 정의서 (솔루션이 아닌 문제) |
| 2 | 왜 지금인가 (트리거) | 트리거 문서화 |
| 3 | 경영진 스폰서십 + 예산 커밋 | 스폰서 지정, 예산 범위 합의 |
| 4 | 성공 KPI 정의 | KPI 목록 (정량 측정 가능해야 함) |
| 5 | [People] 비전 커뮤니케이션 | 조직 공유 메시지 (왜 하는지) |

**산출물: LLM 도입 목적 정의서**

```
1. 비즈니스 문제
   - 현재 상황: [예: 고객 상담 평균 응대 시간 15분, 상담사 매뉴얼 검색에 60% 소요]
   - 해결하고 싶은 것: [예: 상담사 지식 접근 속도 향상 → 응대 시간 단축]
   - 이것이 해결되면: [예: 응대 시간 50% 감소, 고객 만족도 상승]

2. 트리거
   - 왜 지금 시작하는가: [예: 경쟁사 AI 도입, 1년 전 PoC 경험, 기술 성숙]

3. 스폰서십
   - 경영진 스폰서: [이름/직책]
   - 예산 범위: [예: 초기 6개월 API 비용 기준 월 500만원 이내]

4. 성공 KPI
   - [예: 상담 응대 시간 15분 → 7분 (6개월 내)]
   - [예: 상담사 만족도 현재 3.2 → 4.0 (파일럿 종료 시)]

5. 비전 메시지
   - 조직에 공유할 한 줄: [예: "AI가 일을 대체하는 게 아니라, 더 빠르게 답을 찾게 도와줍니다"]
```

## Phase 1: ASSESS — 조직 준비도 평가

**목적:** LLM 도입 전 조직의 현재 상태를 진단한다. 준비 안 된 상태에서 시작하면 실패한다.

> Gartner 예측: 2026년까지 "AI-ready" 데이터 없는 프로젝트의 60%가 폐기
> Kyndryl 2025: 리더의 87%가 AI가 직무를 재편할 것이라 믿지만, 29%만 인력이 준비됐다고 응답

| # | 항목 | 평가 기준 |
|---|------|---------|
| 1 | 데이터 성숙도 | LLM에 넣을 데이터가 있는가, 정리되었는가, 품질은 |
| 2 | 인재/역량 갭 | AI/ML 인력 보유 여부, 운영 인력, 외부 조달 필요성 |
| 3 | IT 인프라 현황 | 클라우드 준비, 네트워크/보안 제약 (망분리, 폐쇄망) |
| 4 | 문화적 준비도 | AI 인식 수준, 저항 예상 영역, 이전 도입 경험 |
| 5 | 규제 환경 사전 스캔 | 적용 규제 식별, 금지/제한 영역, 컴플라이언스 담당 부서 |

**산출물: 조직 준비도 평가서**

```
| 평가 항목 | 현재 수준 | 최소 기준 | 갭 | 조치 |
|---------|---------|---------|---|------|
| 데이터 성숙도 | [예: 문서 있으나 미정리] | 정리된 문서 존재 | 있음 | 문서 정리 선행 (2개월) |
| 인재/역량 | [예: ML 엔지니어 1명] | 최소 운영 가능 인력 | 부분 | 외부 파트너 협업 |
| IT 인프라 | [예: AWS 계정 있음] | 클라우드 기본 구성 | 없음 | 바로 Phase 2 가능 |
| 문화적 준비도 | [예: AI 인식 낮음] | 기본 이해 | 있음 | Phase 4에서 교육 병행 |
| 규제 환경 | [예: 금융권, 금감원 대상] | 적용 규제 식별 완료 | 없음 | Phase 2에서 상세 대응 |
```

> Phase 1 결과가 "데이터 없고, 인력 없고, 인프라 없다"면 Phase 2로 바로 가는 게 아니라 기반 구축부터 한다. 그게 정상이다. 갭이 클수록 Phase 2 이전에 시간이 필요하다.

## Phase 2: GOVERN — 거버넌스 & 정책

**목적:** LLM을 어떤 규칙 안에서 쓸 것인가를 정한다. 이 단계를 대충 넘기면 프로덕션에서 터진다.

> NIST AI RMF: Govern이 첫 번째 함수 — 거버넌스가 전체 프레임워크의 기반
> Gartner 예측: 2026년까지 비인가 AI 사용의 대부분이 외부 공격이 아닌 내부 정책 위반에서 발생할 것

> **Phase 2는 무겁다.** 전부 한꺼번에 하려면 수개월 걸린다. 아래 필수/선택 구분을 따른다.
> - **필수**: 파일럿 전에 반드시 완료. 없으면 Phase 4 진입 불가.
> - **선택**: 파일럿 후 확산(Phase 5) 전까지 완료. 조직 규모와 규제 환경에 따라 조절.

### 2-1. AI 거버넌스 체계 [필수]

| # | 항목 | 설명 |
|---|------|------|
| 1 | 의사결정 기구 | AI 위원회 신설 또는 기존 IT 거버넌스에 편입 |
| 2 | R&R / RACI 정의 | 누가 승인하고, 운영하고, 감사하는가 |
| 3 | AI 인벤토리 관리 | 어떤 AI가 어디서 쓰이는가 추적 체계 |

### 2-2. 보안 정책 [필수]

| # | 항목 |
|---|------|
| 1 | 데이터 분류 (뭘 LLM에 넣어도 되는가) |
| 2 | 접근제어 (누가 어떤 모델을 쓸 수 있는가) |
| 3 | DLP (데이터 유출 방지) |
| 4 | 프롬프트 인젝션 방어 기준 |

> 데이터 분류가 안 되면 Phase 3 PLATFORM에서 뭘 넣고 뭘 못 넣는지 결정할 수 없다. 최소한 데이터 분류(2-2-1)는 반드시 선행.

### 2-3. 규제 대응 [필수 — 범위는 Phase 1 규제 스캔 결과에 따름]

| # | 항목 | 적용 | 필수/선택 |
|---|------|------|:-------:|
| 1 | 인공지능기본법 5대 의무 | 투명성, 안전성, 고영향AI, 영향평가, 국내대리인 (2026.01 시행) | 국내 기업 필수 |
| 2 | 개인정보보호법 | LLM에 개인정보 투입 기준 | 필수 |
| 3 | 금감원 AI 가이드라인 7대 원칙 | 금융사 | 금융사 필수 |
| 4 | 모델 리스크 관리(MRM) | SR 11-7 등 금융 고유 체계에 LLM 통합 | 금융사 필수 |
| 5 | EU AI Act 고위험 분류 | 신용평가, 사기탐지 = 고위험 | 해당 업무 시 필수 |

### 2-4. AI 윤리 원칙 [선택 — Phase 5 전까지]

| # | 항목 |
|---|------|
| 1 | 공정성 / 편향 방지 |
| 2 | 투명성 (AI 사용 고지) |
| 3 | 인간 감독 원칙 (어디까지 자동, 어디서 사람 개입) |

> 파일럿 단계에서는 범위가 좁아 윤리 이슈가 제한적. 확산 전에 확정하면 된다.

### 2-5. 법무 검토 [선택 — Phase 5 전까지]

| # | 항목 |
|---|------|
| 1 | IP / 저작권 (LLM 출력물 소유권) |
| 2 | 계약 검토 (SaaS 이용약관, 데이터 처리 조항) |
| 3 | 책임 소재 (LLM 오답 시 책임) |

> 단, 고객 대면 서비스에 LLM을 쓰는 경우 책임 소재(2-5-3)는 필수로 격상.

### 2-6. 국제 표준 매핑 [선택 — 인증 필요 시]

| 표준 | 목적 | 적용 |
|------|------|------|
| NIST AI RMF | 리스크 관리 체계 정렬 | 글로벌 기업 권장 |

## Phase 4: PILOT — 파일럿 실행

**목적:** 선정된 유즈케이스로 실제 업무에 LLM을 적용하고 검증한다.

### 4-1. 유즈케이스 선정

| # | 항목 |
|---|------|
| 1 | 후보 도출 (각 부서에서 수집) |
| 2 | 우선순위화 (영향도 x 실현가능성 x 리스크) |
| 3 | 현행 프로세스 진단 (As-Is) — 깨진 프로세스에 AI 얹으면 실패 |
| 4 | 파일럿 1~2개 확정 |
| 5 | 유즈케이스 정의서 작성 (아래 양식) |

**산출물: 유즈케이스 정의서** — 이 문서가 IRM의 입력이 된다.

```
1. 도메인
   - 조직/팀: [예: 현대캐피탈 고객상담팀]
   - 업무 범위: [예: 인바운드 상담 — 대출 상품 문의, 상환 문의, 연체 관련]

2. 현행 프로세스 (As-Is)
   - 현재 어떻게 처리하는가: [예: 상담사가 내부 포털에서 매뉴얼 검색 → 고객 응대]
   - 병목 지점: [예: 매뉴얼 검색에 평균 8분 소요, 정확한 문서 못 찾는 경우 30%]
   - 관련 시스템: [예: CRM, 내부 지식포털, 대출 심사 시스템]

3. 기대 효과 (To-Be)
   - LLM이 하는 역할: [예: 고객 질의를 분석 → 관련 매뉴얼 자동 검색 → 상담사에게 추천 답변 제시]

4. 사용자
   - 누가 쓰는가: [예: 고객상담팀 상담사 50명]
   - 파일럿 대상: [예: 대출 상담 파트 10명으로 시작]

5. 리소스 목록 (IRM Phase 0 Step 1 입력)
   - 문서: [예: 대출 상품 매뉴얼, 상환 안내서, 연체 처리 기준, FAQ]

6. 제약 사항
   - 보안: [예: 고객 개인정보 LLM 투입 불가, Phase 2 데이터 분류 기준 적용]
   - 규제: [예: 금감원 AI 가이드라인 — AI 보조수단성 원칙 (최종 판단은 사람)]
   - 인프라: [예: AWS Bedrock 사용, 사내망 내부에서만 접근]
```

### 4-2. IRM 적용

유즈케이스 정의서를 IRM에 넘겨 에이전트 설계를 수행한다.

**핸드오프 매핑:**

| 유즈케이스 정의서 항목 | → IRM 입력 |
|--------------------|-----------|
| 5. 리소스 목록 (문서/시스템/절차/외부) | → IRM Phase 0 Step 1: 리소스 인벤토리 |

> IRM 수행 방법은 IRM 실행가이드 참조. "도메인: ___" 빈칸에 유즈케이스 정의서의 1번(도메인)과 2번(업무 범위)을 넣으면 된다.

## ALFA ↔ IRM 관계

```
ALFA: "LLM을 조직에 어떻게 들여놓을까" (도입 프레임워크)
IRM:  "에이전트한테 일을 어떻게 시킬까" (설계 프레임워크)

ALFA Phase 0~3: LLM 도입 준비
ALFA Phase 4:   유즈케이스 선정 → ★ IRM 투입 ★ → 파일럿
ALFA Phase 5~6: 확산 + 운영 (IRM 매핑표 계속 축적)
```

ALFA 없이 IRM만 쓰면: "어떤 업무에 쓸지"가 정해지지 않은 채 설계를 시작하게 된다.
IRM 없이 ALFA만 쓰면: "도입하기로 했는데 구체적으로 어떻게 만들지"가 없다.

│   ├── AI 거버넌스 체계 (의사결정, R&R, AI 인벤토리)

│   └── 법무 검토
