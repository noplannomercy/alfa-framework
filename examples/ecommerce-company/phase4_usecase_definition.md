# Phase 4: 유즈케이스 정의서 — CS 상담사 보조

> 고객: 이커머스 기업
> Phase: 4
> 상태: 완료
> 이 문서가 IRM의 입력이 된다.

---

## 1. 도메인

- **조직/팀:** CS팀 (상담사 15명)
- **업무 범위:** 고객 문의 응대 — 배송 조회, 교환/환불, 상품 문의

---

## 2. 현행 프로세스 (As-Is)

- **현재:**
  1. 고객 문의 접수 (전화/채팅/이메일)
  2. FAQ 페이지 확인 (오래됨, 잘 안 씀)
  3. 교환/환불 정책 문서 검색 (공유폴더)
  4. 상품 정보 확인 (쇼핑몰 어드민)
  5. 배송 상태 조회 (택배사 API)
  6. 모르면 선임한테 슬랙으로 물어봄
  7. 고객에게 답변

- **병목:**
  - 4곳에 분산된 정보를 각각 뒤져야 함
  - 교환/환불 정책이 복잡 (개봉 여부, 할인 상품, 기간 등 케이스마다 다름)
  - 하루 500건 / 15명 = 1인당 33건. 1건당 10분이면 하루 5.5시간
  - 반복 질문 60%+

- **관련 시스템:**
  - FAQ 페이지 (웹)
  - 교환/환불 정책 문서 (공유폴더, 30페이지)
  - 쇼핑몰 어드민 (상품 정보, SKU 2000개)
  - 택배사 API (배송 조회)

---

## 3. 기대 효과 (To-Be)

- **LLM이 하는 역할:**
  - 고객 질의 → 관련 FAQ + 정책 조항 + 상품 정보 자동 검색
  - 상담사에게 추천 답변 제시 (상담사가 확인 후 발송)
  - 복잡한 교환/환불 케이스도 정책 문서에서 해당 조항 바로 찾아줌

- **KPI:**
  - 상담사 1건당 응대 시간: 10분 → 5분
  - 선임 문의 비율: 현재 대비 50% 감소
  - 반복 질문 자동 추천 정확도: 80% 이상

---

## 4. 사용자

- **누가 쓰는가:** CS팀 상담사 15명
- **파일럿 대상:** 15명 전원 (소규모팀)
- **2차:** 고객 대면 챗봇 전환 (거버넌스 강화 후)

## 6. 제약 사항

- **보안:** 상담사 보조 단계에서는 고객 개인정보 LLM 투입 없음 (질의 내용만)
- **인프라:** AWS Bedrock. 벡터DB 별도 필요 (MySQL → pgvector 불가)
- **예산:** 월 300만원
- **고객 대면:** 2차. 전환 시 오답 책임 + 개인정보 처리 검토 필요

## IRM 수행 예측

```
IRM Phase 0:
  - 행위 유형: 탐색(내부) 중심 + 접근(어드민, 택배 API)
  - 질의 패턴: "교환 되나요?", "배송 어디까지 왔어요?", "이 상품 재고 있어요?"

IRM Phase 1:
  - 1-A 탐색: RAG (FAQ + 정책 문서 + 배송 안내)
  - GraphRAG: 불필요 (조건 분기, 인과 아님)
  - 1-B 접근: 어드민 DB (상품 정보), 택배 API (배송 조회)
  - 1-C 실행(확정): 없음 (상담사 보조는 검색)
  - 1-D 실행(유연): 없음

IRM Phase 2:
  - RAG §1~§8 (소량 데이터, 기본 구성)
  - CLI: 어드민 DB, 택배 API
```
