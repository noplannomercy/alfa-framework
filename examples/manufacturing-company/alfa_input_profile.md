## [3] 인프라 현황

- **클라우드:** 없음 (온프레미스 전용)

- **기존 DB:** PostgreSQL (품질 DB, 자체 개발, MES 연동)

- **문서 저장:** 품질 포털 (PDF), 공유폴더 (Word), 설비팀 서버

## [4] 규제 환경

- **업종 규제:** 가벼움 (금융/의료 수준 아님)
- **보안 정책:** 데이터 외부 반출 금지 (회사 정책)
- **인증:** ISO 9001 / IATF 16949 여부 미확인 (확인 필요)
- **데이터 분류:** 전부 사내 문서 + 사내 품질 데이터. 개인정보 없음
- **컴플라이언스 별도 검토:** 불필요

## [5] 하고 싶은 것

- **스폰서:** 공장장 (지시 — "불량 대응 체계 개선해라, AI로 할 수 있으면 해봐라")
- **현업 니즈:** 불량 원인 분석 속도 개선
  - 현재: 불량 발생 → 검사 기준서/과거 이력/공정 매뉴얼 각각 뒤짐 → 박부장님한테 물어봄
  - 병목: 간단한 불량 1~2시간, 복잡한 불량 하루+, 최악 3일
- **1차 파일럿:** 품질 문서 + 불량 이력 통합 검색 (품질팀 15명)
- **2차 확장:** 생산팀 30명으로 사용자 확대
- **기술 이슈:** 같은 불량인데 표현이 다름 ("버 발생" vs "성형 불량") → GraphRAG 시그널

---

## [6] 제약 사항

- **예산:** 추가 장비 구매 불가. 기존 A100 + PostgreSQL 활용
- **일정:** 3개월 (6월까지, 공장장 기대)
- **내부 인력:** 최차장 (품질+IT 겸임)
- **외부 파트너:** 아키텍트 (설계+구축)
- **인프라 제약:** 온프레미스 필수, 외부 API 불가

## 기술 메모 (내부)

- **GraphRAG 승격 시그널 강함:** 동음이의어 ("버 발생"/"성형 불량") + 인과 체인 (불량→원인→공정→설비)
- **IRM 연결:** `docs/IRM/hist/manufacturing-qc/` 시뮬레이션 직접 참조 가능
- **로컬 스택:** Ollama + PostgreSQL (pgvector + Apache AGE) + BGE-m3-ko
- **불량 이력 만 건:** 텍스트 필드(증상/원인/조치) RAG 인덱싱 + 구조화 필드 메타데이터
