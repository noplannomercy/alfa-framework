# Phase 4: 유즈케이스 정의서 — 품질 불량 원인 분석

> 고객: 제조기업 (자동차 부품)
> Phase: 4-1 ~ 4-2
> 상태: 완료
> 이 문서가 IRM의 입력이 된다.

---

## 1. 도메인

- **조직/팀:** 품질관리팀 (15명)
- **업무 범위:** 불량 발생 시 원인 분석, 유사 사례 검색, 검사 기준 확인, 공정 매뉴얼 참조

---

## 2. 현행 프로세스 (As-Is)

- **현재 어떻게 처리하는가:**
  1. 불량 발생 보고 (MES → 품질 DB)
  2. 검사 기준서에서 해당 불량 기준 확인 (품질 포털 PDF 검색)
  3. 과거 유사 불량 이력 조회 (품질 DB → 엑셀 추출 → 수동 검색)
  4. 공정 매뉴얼에서 관련 공정 조건 확인 (공유폴더 Word 검색)
  5. 설비 이력 확인 (설비팀 서버)
  6. 박부장님한테 물어봄
  7. 원인 특정 → 조치

- **병목 지점:**
  - 4곳에 분산된 문서를 각각 뒤져야 함
  - 품질 DB 검색이 키워드 기반이라 같은 불량인데 표현이 다르면 못 찾음 ("버 발생" vs "성형 불량")
  - 결국 박부장님(경력 20년)에게 의존 → 속인화
  - 간단한 불량 1~2시간, 복잡한 불량 하루+, 최악 3일

- **관련 시스템:**
  - 품질 포털 (검사 기준서 PDF)
  - 품질 DB (PostgreSQL, MES 연동, 불량 이력)
  - 공유폴더 (공정 매뉴얼 Word)
  - 설비팀 서버 (설비 이력/자료)
  - MES (공정 데이터)

---

## 3. 기대 효과 (To-Be)

- **LLM이 하는 역할:**
  - 자연어 질의 → 검사 기준서 + 불량 이력 + 공정 매뉴얼을 한번에 검색
  - 유사 불량 사례를 의미 기반으로 찾아줌 ("버 발생"과 "성형 불량"이 같은 건지 판단)
  - 불량→원인→공정→설비 관계 체인 추적 (GraphRAG 승격 시)

- **Phase 0 KPI 연결:**
  - 복잡 불량 원인 분석: 하루+ → 2~3시간
  - 유사 사례 검색: 수동(박부장 의존) → 시스템 검색
  - 박부장 의존도: 높음 → 시스템 대체 가능

## 6. 제약 사항

- **보안:** 온프레미스 필수. 데이터 외부 반출 금지. 외부 API 사용 불가
- **인프라:** A100 서버 + PostgreSQL (기존). Ollama + pgvector + Apache AGE
- **예산:** 추가 장비 없음. 소프트웨어 설치만
- **규제:** 업종 규제 가벼움. 컴플라이언스 검토 불필요

## IRM 수행 예측

```
IRM Phase 0:
  - 행위 유형: 탐색(내부) 중심 + 접근(품질DB, MES)
  - 질의 패턴: "비슷한 불량 있었어?", "이 불량 원인이 뭐야?", "검사 기준 알려줘"

IRM Phase 1:
  - 1-A 탐색: RAG 대상 (문서 150개 + 불량 이력 텍스트)
  - GraphRAG 승격 판단: 동음이의어 + 인과 체인 → 승격 대상 (3/4 충족 예상)
  - 1-B 접근: 품질 DB, MES 조회
  - 1-C 실행(확정): 검사 절차 (수입/공정/출하) — 해당 시
  - 1-D 실행(유연): 불량 원인 분석, CAPA — 해당 시

IRM Phase 2:
  - RAG 설계방법론 §1~§8
  - GraphRAG 설계방법론 §1~§4 (승격 시)
  - CLI 런북 (품질DB, MES 접근)
```

> **참조:** IRM hist `manufacturing-qc` 시뮬레이션이 이 도메인과 직접 대응된다.
> IRM Phase 0~2의 상세 결과는 해당 시뮬레이션 참조 가능.
