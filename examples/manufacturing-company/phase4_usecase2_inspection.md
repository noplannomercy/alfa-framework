# Phase 4: 유즈케이스 정의서 #2 — 검사 절차 자동화

> 고객: 제조기업 (자동차 부품)
> Phase: 4 (2차 유즈케이스)
> 상태: 완료
> 이 문서가 IRM의 입력이 된다.

---

## 1. 도메인

- **조직/팀:** 품질관리팀 (15명)
- **업무 범위:** 수입검사, 공정검사, 출하검사 절차 실행

---

## 2. 현행 프로세스 (As-Is)

- **수입검사:**
  1. 자재 입고 확인 (ERP)
  2. 검사 기준서에서 해당 자재 검사 항목 확인 (PDF 수동 검색)
  3. 시험 수행 또는 LIMS 요청
  4. 합격/불합격 판정 (기준 대조)
  5. 품질 DB에 결과 기록 (수동 입력)

- **공정검사:**
  1. MES에서 공정 데이터 확인
  2. 검사 기준서 확인
  3. SPC 관리도 확인
  4. 합격/불합격 판정
  5. 품질 DB 기록

- **출하검사:**
  1. 품질 DB에서 로트별 검사 이력 확인
  2. 최종 시험 수행
  3. 출하 판정 (합격/조건부/불합격)
  4. 검사 성적서 발행

- **병목:**
  - 각 단계마다 다른 시스템 접근 (ERP, MES, 품질 DB, LIMS)
  - 기준서 수동 검색
  - 결과 수동 기록
  - 순서가 고정되어 있는데 사람이 일일이 수행

- **관련 시스템:**
  - ERP (자재 입고)
  - MES (공정 데이터)
  - 품질 DB (검사 결과 기록, PostgreSQL)
  - LIMS (시험 요청/결과) — 있는지 확인 필요
  - 검사 기준서 (PDF, 품질 포털)

---

## 3. 기대 효과 (To-Be)

- **LLM/CLI가 하는 역할:**
  - 검사 절차를 CLI 파이프라인으로 자동 실행
  - 기준서 자동 검색 (RAG — 1차 유즈케이스 인프라 재활용)
  - 합격/불합격 판정 자동 대조
  - 결과 자동 기록

- **KPI:**
  - 검사 1건 소요 시간 단축 (수동 30분 → 자동 5분)
  - 기록 누락/오류 제거

---

## 6. 제약 사항

- **보안:** 온프레미스 필수 (기존과 동일)
- **인프라:** 기존 A100 + PostgreSQL 재활용. RAG 인프라 이미 있음
- **규제:** 업종 규제 가벼움 (기존과 동일)
- **추가:** ERP, LIMS 접근 권한 확인 필요 (1차에는 없었던 시스템)

## IRM 수행 예측

```
1차 유즈케이스 (불량 원인 분석):
  - 탐색 중심, GraphRAG 필수, 실행(확정) 없음

2차 유즈케이스 (검사 절차 자동화):
  - 실행(확정) 중심 ← 완전히 다름
  - CLI 파이프라인형 (quality, mes, erp CLI)
  - 탐색은 기준서 룩업 (기존 RAG 재활용)
  - GraphRAG 불필요
```

> **같은 고객, 같은 도메인인데 IRM 결과가 완전히 달라진다.**
> 이게 유즈케이스별로 IRM을 돌리는 이유.
