# 1차 미팅 대화 기록

> 일시: 2026-03-25
> 참석: 최차장 (품질관리팀, IT 겸임), 아키텍트

---

**최차장:** 안녕하세요. 저희가 품질 쪽에서 AI를 좀 써보고 싶은데요. 불량 나면 원인 찾는 데 시간이 너무 걸려요. 검사 기준서, 과거 불량 이력, 공정 매뉴얼 이런 게 다 따로따로 있어서 하나하나 뒤져야 하거든요. LLM으로 이걸 한번에 찾을 수 있다고 들어서요.

**아키텍트:** 제조 품질 쪽에서 그 문제 많이 들어요. 문서가 흩어져 있으면 경험 많은 사람한테 물어보게 되죠. 오늘은 현재 상황부터 파악하고 방향 잡아드릴게요.

**최차장:** 맞아요, 결국 경력 20년 된 박부장님한테 물어봐요. 그분이 휴가 가면 아무도 모르는 거예요. ㅋㅋ 이게 속인화 문제라서... 문서로 다 있긴 한데 찾는 게 문제예요.

**아키텍트:** 속인화 문제 — 정확히 그거예요. 박부장님 머릿속에 있는 걸 시스템으로 옮기는 거죠. 그 문서들이 지금 어디에 있어요? 사내 시스템이요, 공유폴더요?

**최차장:** 여기저기 흩어져 있어요. 검사 기준서는 품질 포털에 PDF로 있고, 불량 이력은 품질 DB에 있는데 엑셀로 뽑아서 보고, 공정 매뉴얼은 공유폴더에 워드 파일로... 아 그리고 설비 관련 자료는 설비팀 서버에 따로 있어요.

**아키텍트:** 4곳에 흩어져 있으면 그게 문제 맞아요. 한 가지씩 볼게요 — 문서가 전체 몇 건 정도 되고, 품질 DB는 어떤 시스템이에요?

**최차장:** 검사 기준서가 한 100개, 공정 매뉴얼이 50개 정도, 불량 이력은 품질 DB에 한 3년치 쌓여있어요. 건수로는 만 건 넘을 거예요. 품질 DB는 자체 개발한 거예요. PostgreSQL 기반이고 MES에서 불량 데이터 받아서 쌓아요.

**아키텍트:** 문서 150개 + 불량 이력 만 건. 인프라 궁금한데 — 지금 클라우드 쓰세요, 온프레미스예요? 그리고 불량 이력에 증상이나 원인 같은 텍스트 필드가 있어요?

**최차장:** 클라우드는 아직이에요. 전부 온프레미스. 공장이다 보니까 보안 때문에 외부 연결을 꺼려해요. 불량 이력에는 증상, 원인, 조치 내용이 텍스트로 있어요. 근데 사람마다 쓰는 방식이 달라서 "버 발생"이라고 쓰는 사람도 있고 "성형 불량"이라고 쓰는 사람도 있고... 같은 불량인데 표현이 달라요.

**아키텍트:** 같은 불량인데 표현이 다르다 — 그건 나중에 기술적으로 잡을 수 있어요. 메모해둘게요. 온프레미스라고 하셨는데, 혹시 규제나 보안 정책이 따로 있어요? 데이터 외부 반출 금지 같은 거요.

**최차장:** 공장 데이터는 외부 반출 금지예요. 회사 보안 정책상 클라우드에 올리는 것도 윗분들이 꺼려해요. 그래서 사내 서버에서 다 돌려야 하는데... 그게 가능해요?

**아키텍트:** 가능해요. 요즘 로컬에서 돌리는 LLM이 성능이 많이 올라왔어요. 사내 서버에 GPU만 있으면 됩니다. 비용은 나중에 보고요 — 먼저 다른 것 좀 더 확인할게요.

**최차장:** 아 다행이다. GPU 서버는... 있긴 한데 학습용으로 하나 사놓은 게 있어요. A100인가? 거의 안 쓰고 있어요.

**아키텍트:** A100 놀고 있으면 그대로 쓰면 돼요. 외부 API 비용도 없고 GPU도 있으니 인프라는 해결이네요. 나머지 확인할게요 — 이 프로젝트 위에서 지시한 건가요?

**최차장:** 공장장님이 지시하셨어요. 작년에 불량 대응 늦어서 고객 클레임 크게 한번 터졌거든요. 그때 "불량 대응 체계를 개선해라" 하셨는데, AI로 해결할 수 있으면 해보라고요. 예산은... 솔직히 많지 않아요. 추가 장비 사는 건 어려운데 있는 거 활용하는 건 괜찮다고 하셨어요.

**아키텍트:** 있는 거 활용이면 딱 맞아요. A100 있고, PostgreSQL 있고, 추가 비용은 거의 없어요. 이걸 누가 쓸 거예요? 품질팀만이요, 생산팀도?

**최차장:** 품질팀이 주로 쓸 거예요. 한 15명 정도. 근데 생산팀도 공정 이상 나면 매뉴얼 찾아봐야 하니까 나중에는 쓰고 싶어할 거예요. 생산팀이 한 30명이고요.

**아키텍트:** 좋아요. 하나만 더 — 지금 불량 나면 원인 찾는 데 보통 얼마나 걸려요? 이게 파일럿 성공 기준이 되거든요.

**최차장:** 불량 원인 찾는 데요... 간단한 거는 1~2시간이면 되는데, 복잡한 거는 하루 이상 걸려요. 과거 유사 사례 뒤지고, 공정 조건 확인하고, 설비 이력 보고... 그때마다 박부장님 붙잡고요. 작년 클레임 건은 원인 찾는 데만 3일 걸렸어요.

**아키텍트:** 복잡한 건 하루 이상, 심하면 3일 — 이걸 2~3시간으로 줄이는 게 목표가 되겠네요. 일정은 어떻게 보세요? 공장장님이 언제까지 뭔가 보고 싶어하세요?

**최차장:** 공장장님이 올해 상반기 안에는 뭔가 보여달라고 하셨어요. 지금 3월이니까... 6월까지? 3개월 정도요.

**아키텍트:** 됩니다. 정리합니다 — 온프레미스 A100 서버에 LLM + PostgreSQL pgvector. 품질팀 15명 대상으로 불량 원인 분석 검색 파일럿. 문서 150개 + 불량 이력 만 건. 목표는 복잡 불량 분석 하루 → 2~3시간. 3개월이면 결과 나옵니다. 이번 주 안에 상세 일정 보내드릴게요.

**최차장:** 오 깔끔하네요. 근데 한 가지 — 아까 말한 같은 불량인데 표현이 다른 문제, 그건 해결이 돼요?

**아키텍트:** 됩니다. '버 발생'이랑 '성형 불량'이 같은 건지 시스템이 판단하게 할 수 있어요. 기술적으로 그래프 기반 관계 탐색이라는 게 있는데, 파일럿 돌려보고 그 문제가 심하면 적용합니다. 지금은 일단 기본 검색부터 깔고요.

**최차장:** 아 그렇군요. 좋습니다. 바로 시작하죠.

**아키텍트:** 최차장님은 A100 서버 접근 권한이랑, 품질 DB 조회 권한만 확인해 주세요. 그리고 검사 기준서 100개, 공정 매뉴얼 50개 파일 위치만 정리해서 공유해 주시면 됩니다.

**최차장:** 네, 바로 정리해서 보내드릴게요.
